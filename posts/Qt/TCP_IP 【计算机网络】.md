FIN-值为1表示数据报文已经发送完毕，需要释放连接。

窗口大小：限制发送方的窗口上限。

一、TCP和UDP的区别与联系，都有那些应用用到了TCP和UDP协议，各自数据包的特点？

1、tcp基于请求确认机制。

2、网络协议经过确认可以百分百确认可靠性，但最新发出的报文永远无法确认。

3、tcp保证数据的按序到达。

4、有丢包重传机制(确认序号对不上)。

5、tcp面向连接。

6、tcp可以进行流量控制(窗口大小表示接收缓冲区大小)。

7、tcp提供各种定时器(超时重传计时器)。

8、tcp实现网络拥塞避免算法。

# 2、TCP与UDP区别总结：
1、TCP面向连接（如打电话要先拨号建立连接）;UDP是无连接的，即发送数据之前不需要建立连接。

2、TCP提供可靠的服务。也就是说，通过TCP连接传送的数据，无差错，不丢失，不重复，且按序到达;UDP尽最大努力交付，即不保证可靠交付。

3、TCP面向字节流，实际上是TCP把数据看成一连串无结构的字节流;UDP是面向报文的UDP没有拥塞控制，因此网络出现拥塞不会使源主机的发送速率降低（对实时应用很有用，如IP电话，实时视频会议等）

4、每一条TCP连接只能是点到点的;UDP支持一对一，一对多，多对一和多对多的交互通信

5、TCP首部开销20字节;UDP的首部开销小，只有8个字节

6、TCP的逻辑通信信道是全双工的可靠信道，UDP则是不可靠信道

# 3、TCP如何实现可靠性
TCP发送的报文段是交付给IP层的，但是IP层只能提供最大能力交付，即tcp下面的网络所提供的服务是不可靠的，因此要提供可靠性，只能tcp在传输层采取适当措施。

采取措施：

## a、停止等待协议：
无差错：发送发每发送一个分组就停止发送，等待对方的确认，在收到确认后，再发送下一个分组

有差错：若接收一个报文出现差错(也可能是发送方发送报文出现差错)，就丢弃改报文，不做任何事，此时发送方超过一定时间没有收到对方确认，当即重传前面发送过的分组，这就是超时重传。（发送方每发送一个分组都设置了一个超时计时器(tcp采用一种自适应算法，自行决定超时重传的时间：记录一个报文段发出时间，以及收到确认的时间，时间之差RTT,公式计算重传时间)）。

## b、TCP流量控制：
流量控制就是让发送发发送的发送速率不要太快，要让接收方来得及接收。发送方的发送窗口不能不能超过接收方给出的接受窗口的大小(注：窗口单位是字节，不是报文段)

## c、拥塞避免：
拥塞：若网络中某一资源的需求超过了该资源所能提供的可用部分，网络的性能变坏，即表示网络拥塞。(例如：只有5个资源，而有10个取资源，本来不够后面读取还一直增加),(只要发送方没有按时收到应当到达的报文，就猜想网络可能出现拥塞)。

拥塞控制：就是防止过多的数据注入到网络，是路由器或链路不至于过载。

拥塞控制方法：

慢启动：当主机发送数据时，它不是立即把大量数据注入到网络中(因为可能出现网络拥塞)，一开始并不清楚网络的负载情况，因此它采用一种试探的方法，即就是由小到大的增大发送窗口，当新建连接时，cwnd初始化为1个最大报文段(MSS)大小，发送端开始按照拥塞窗口大小发送数据，每当有一个报文段被确认，cwnd就增加1个MSS大小。这样cwnd的值就随着网络往返时间(RTT)呈指数级增长。但是这样增长很快，为了防止拥塞窗口增长过大，引起网络拥塞，它还设置一个门限值ssthresh。

当cwnd < ssthresh : 继续使用刚那个慢启动算法；

当cwnd > ssthresh : 停止使用慢启动而使用拥塞避免算法；

当cwnd = ssthresh : 两个任意用一个。

拥塞避免算法：刚开始让成指数形式增加太快了，为了防止拥塞，它采用经过一个往返时间，它不是增加一倍了，而是在它基础上加1个mss报文长度，这样是拥塞窗口按线性增长速率增长。

注意：无论是在慢启动还是拥塞避免阶段，只要发送方判断网络出现拥塞，就立即把sshthre减小一般，并把拥塞窗口重新设置为1，执行慢启动算法。

 快重传：快重传算法首先要求接收方每收到一个失序的报文段就立即发出重复确认，为了告诉发送方有报文段没有到达对方，快重传算法规定，只要一连收到3个重复确认就应当重传对方尚未收到的报文段。(停止等待协议不一定非的是发一条确认一条，也有可能是发送多个再确认以提高效率)。

 快恢复：与快重传配合使用，当收到三个重复确认后，将窗口大小设置为(ssthresh)的一半。然后执行拥塞避免算法。

# 4、UDP如何实现可靠性传输
UDP不属于连接协议，具有资源消耗少，处理速度快的优点，所以通常音频，视频和普通数据在传送时，使用UDP较多，因为即使丢失少量的包，也不会对接受结果产生较大的影响。

传输层无法保证数据的可靠传输，只能通过应用层来实现了。实现的方式可以参照tcp可靠性传输的方式，只是实现不在传输层，实现转移到了应用层。

最简单的方式是在应用层模仿传输层TCP的可靠性传输。下面不考虑拥塞处理，可靠UDP的简单设计。

+ 1、添加seq/ack机制，确保数据发送到对端。
+ 2、添加发送和接收缓冲区，主要是用户超时重传。
+ 3、添加超时重传机制。

详细说明：送端发送数据时，生成一个随机seq=x，然后每一片按照数据大小分配seq。数据到达接收端后接收端放入缓存，并发送一个ack=x的包，表示对方已经收到了数据。发送端收到了ack包后，删除缓冲区对应的数据。时间到后，定时任务检查是否需要重传数据。

# 5、TCP黏包
1、什么是TCP黏包：

因为TCP是面向字节流的协议，没有数据边界，导致接收侧有可能一下子收到多个应用层报文，需要应用开发者自己分开。

2、UDP丢失、乱序解决办法

丢失：

服务器端的socket接收缓存满了(**udp**没有流量控制，因此发送速度比接收速度快，很容易出现这种情况)，然后系统就会将后来收到的包丢弃。可以尝试用setsockopt()将接收缓存(SO_RCVBUF)加大看看能不能解决问题。

乱序：

可以采用发送端在数据段中加入数据报序号的方法，这样接收端对接收到数据的头端进行简单地处理就可以重新获得原始顺序的数据
